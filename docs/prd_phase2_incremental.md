# SkyChart Pro 二期增量 PRD（简单版）

> 作者：许清楚（Xu）｜ 产品经理
> 版本：v2.0-incremental ｜ 配套：一期 `docs/system_design.md`
> 说明：本 PRD 仅描述二期**新增/变更**内容，不重写一期既有功能。编号沿用 A/B/C/D/E/F 模块。
> 技术约束：保持 vanilla JS + `<script>` 全局作用域 + 单数据源 + 事件委托 + localStorage try/catch + 手绘 SVG/CSS；禁止 React/Vue/Vite/打包器/图表库。新状态只在 `state.js`、新 DOM 引用只在 `dom.js`、新交互只在 `events.js` 委托、初始化插 `main.js init`、新脚本在 `main.js` 之前按序追加。

---

## 1. 产品目标

**一句话目标**：把 SkyChart Pro 从"本地硬编码演示"升级为"可联网真实读取航图、可中英切换、可下载、界面更精致且署明数据来源的公开产品"。

**3 条关键价值**：
1. **真实可用**：直连 `chart.wuhanqing.cn` 远程航图，解决部署后 404，让 GitHub Pages / CloudStudio 用户真正看到航图。
2. **开放与合规**：明确数据来源署名（主源 + JeppView + EAIP 版本 + 下载全部），并公示维护者，便于社区共建与版权合规。
3. **体验升级**：中英切换、航图下载、动态背景/类型彩标/快捷键/随机发现等吸睛增强，提升专业感与探索乐趣。

## 2. 用户故事

### A. 数据来源接入 + 署名
- 作为部署用户，我希望打开网页就能看到真实航图 PDF（而非 404 白屏），以便实际使用本工具。
- 作为使用者，我希望清楚知道航图来自哪里、版本号是多少，以便引用与核验。
- 作为遇到加载失败的用户，我希望看到友好提示并能跳去源站或下载整包，而非卡在白屏。

### B. 中英文切换
- 作为英文用户，我希望界面切换为英文，以便我理解与使用。
- 作为切换者，我希望点击按钮即时切换、不刷新页面，保持流畅。

### C. 航图下载
- 作为研究者，我希望一键下载某张航图到本地，便于离线查看。
- 作为机场管理员，我希望批量下载某机场全部航图，便于整体备份。

### D. 吸睛功能
- 作为访客，我希望背景/标签/动效更精致有科技感，以便留下好印象。
- 作为键盘党，我希望有快捷键帮助面板，以便高效操作。
- 作为探索者，我希望有"随机航图"入口，以便发现有趣内容。

### E. 页脚维护者信息
- 作为潜在贡献者，我希望在页脚看到维护者 GitHub 与邮箱，以便联系或提 PR。
- 作为读者，我希望数据来源与维护者信息层次清晰，不混淆。

### F. 界面精修
- 作为用户，我希望空/加载/错误态与微交互统一美观，以便整体质感一致。

## 5. 待确认问题（附推荐方案）

1. **A-01 前缀替换的落地方式**：
   - 方案①（推荐）：保持 `charts-data.js` 的 `path` 不变（仍是 `航图/...`），在 `buildChartUrl` 内做"前缀映射 + 分段编码"，并对绝对 URL 只编码 origin 之后部分。优点：单一数据源、避免 1.3MB 大文件重写风险、本地 `file://` 调试仍可回退、远程基址一处可配。
   - 方案②：一次性脚本把 `charts-data.js` 全部 `航图/` 替换为 `https://chart.wuhanqing.cn/Terminal/`，并同步改造 `buildChartUrl` 处理绝对 URL。优点：数据即最终态。
   - **无论哪种，`buildChartUrl` 都必须处理绝对 URL 的编码**，否则 `https://` 被 encode 成 `https%3A//` 而失效。

2. **C-01/C-02 跨域下载限制**：跨域 PDF 的 `<a download>` 会被浏览器忽略 download 属性（仅导航/打开），真正"下载并重命名"需 `fetch+blob`（依赖 `chart.wuhanqing.cn` 开启 CORS）。建议：默认用 `download` 属性 + 新标签兜底；是否需要我方向源站确认 CORS 头以便做 fetch+blob 真下载？（默认不做 CORS 依赖，走兜底。）

3. **C-02 批量下载的浏览器限制**：顺序 `click()` 触发多下载常被浏览器拦截（仅首个通过）。建议：批量按钮优先引导至"下载全部包"(r2 zip)，并在本机场场景用"依次触发 + 提示用户允许多次下载"的轻量方案，不引入打包/压缩库。

4. **A-03 远程失败判定**：iframe PDF 加载失败难以精准捕获（无可靠 onerror）。建议：复用现有 8s `modalLoadTimer` 思路——打开后超时仍空白则显示兜底；或先 `fetch(head)` 探活。请确认采用"超时兜底"还是"预检 HEAD"（推荐超时兜底，零额外请求）。

5. **B i18n 覆盖范围**：是否覆盖全部一期文本（含仪表盘所有标签、我的区、抽屉说明、空/加载态），还是仅主界面与导航？推荐全量覆盖以保证中英一致；工作量主要为补 `I18N` 字典与 `t()` 化现有模板。需确认是否接受全量。

6. **D-01 性能取舍**：动态背景是否允许轻量 canvas 粒子（更炫但更耗电），还是纯 CSS 动画（更省）。建议默认纯 CSS 极光 + `prefers-reduced-motion` 降级；若要做粒子，限定低帧率/小画布。

7. **E-01 邮箱口径**：原页脚 `support@/contact@skychartpro.top` 是否直接替换为 `zhang429500@icloud.com`（按用户"邮箱全部统一"），还是保留为产品邮箱、另加维护者邮箱？推荐按用户要求统一替换为 `zhang429500@icloud.com`，数据来源署名块保留原作者邮箱不变。

8. **现有免责声明更正**：当前页脚写"本站航图内容均来源于Jeppesen航图"与二期调研（主源 chart.wuhanqing.cn、部分 JeppView）不符。建议在 A-02/E-02 中一并更正为客观来源描述。请确认是否重写该句。
